# Analyze Diagnostics Reports

Use this skill to analyze single-file Diagnostics reports. The report may be a newer JSON-backed HTML file or an older HTML-only report.

## First Choice: Embedded JSON

When the file contains this element, treat it as the source of truth:

```html
<script id="diagnostics-report-data" type="application/json">
```

Extract and parse the JSON from that script tag. Ignore CSS, JavaScript, navigation markup, and rendered HTML when structured JSON is present. Those sections are for browser display and waste tokens.

Expected top-level fields:

- `schemaVersion`: report schema version.
- `title`: human report title.
- `agentHints`: producer-provided analysis hints.
- `chapters`: report sections.

Expected chapter shapes:

- `kind: "logs"` with `data.sessions`.
- `kind: "table"` with `data.rows`.
- `kind: "text"` or `kind: "preformatted"` with `data.value`.
- `kind: "legacyHTML"` for custom or old fallback content.

## Legacy HTML Fallback

If no JSON script exists, parse the HTML report directly:

- Session logs are usually inside `#log-sessions`.
- Sessions are usually wrapped in `.collapsible-session details`.
- Debug, error, and system logs usually use `.debug`, `.error`, and `.system`.
- Older persisted sessions may contain a `<summary><div class="session-header">...` block followed by log paragraphs.

Only use raw rendered HTML when JSON is unavailable or a chapter is explicitly marked `legacyHTML`.

## Analysis Order

Analyze in this order:

1. Read `agentHints` if present.
2. Inspect app/system metadata for version, OS, locale, device, and build context.
3. Inspect newest log sessions first.
4. Prioritize `error` events, crashes, MetricKit diagnostics, and unusual system logs.
5. Compare the failing session with older sessions when useful.
6. Check custom chapters for domain-specific state, feature flags, user defaults, directory trees, or configuration clues.

## Output Style

Lead with the likely issue and evidence. Keep the output concise and useful for a developer or support engineer.

Use this structure unless the user asks for something else:

```markdown
## Summary
One or two sentences describing what most likely happened.

## Evidence
- Specific session, timestamp, event level, or chapter detail that supports the conclusion.
- Another relevant detail if needed.

## Recommended Next Steps
- Concrete debugging or support action.
- Code/config area to inspect if apparent.
```

Avoid dumping raw logs. Quote short snippets only when they directly support a conclusion.

## Token Discipline

Diagnostics reports can be large. Do not load or restate the whole HTML file when JSON exists. Extract the JSON payload, then narrow to:

- newest sessions
- error/crash events
- metadata rows
- custom chapters related to the user's question

If logs are noisy, summarize counts and patterns before quoting individual entries.
